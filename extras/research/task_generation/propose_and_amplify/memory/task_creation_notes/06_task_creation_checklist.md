# Task Creation Checklist

## Overview

Use this checklist for every task you create. Check off items as you complete them.

---

## Phase 1: Research & Design

### Look Up Occupation/Industry Context (REQUIRED BEFORE ANYTHING ELSE)
- [ ] Ran the occupation lookup (Step 1 & 2 in `07_agent_prompt_template.md §0`) for this product
- [ ] Identified the top 5 occupations by `product_gdp_usd`
- [ ] Read `category_rationale` for each top occupation
- [ ] Can state in one sentence: *"Real [occupation] professionals use this software to [do X]"*
- [ ] Each task candidate reflects a workflow that a top-occupation user would recognise as realistic

> If the product is absent from `master_dataset.csv`/`selected_products.csv`, note that and proceed without this step.

### Understand the Environment
- [ ] Environment is running and accessible
- [ ] Know how to query the database (if applicable)
- [ ] Understand the application's main workflows AND its less-obvious features
- [ ] Do NOT use existing tasks as a difficulty reference — many existing tasks are too simple

### Identify Task Candidate
- [ ] Task represents real professional workflow
- [ ] Task is achievable in the environment
- [ ] Litmus test: a power user who has never used this software could NOT solve this in under 10 minutes
- [ ] Task requires 3+ distinct application features, not a single workflow
- [ ] Agent must discover what to do (not just be told which record to find and what value to enter)
- [ ] Complexity matches intended difficulty level
- [ ] Task has clear, verifiable success criteria (unambiguous goal, not necessarily spelled-out path)
- [ ] Starting state is distinct from existing tasks (different records/data, not just a different action on the same dataset)

### Verify ALL Data Is Real (MANDATORY)
- [ ] ALL data used by the task is real — sourced from the environment's database, real public datasets, or sample data bundled with the software
- [ ] NO synthetic, generated, simulated, or fabricated data anywhere — not in setup_task.sh, not in data/ directories, not anywhere
- [ ] If the task needs input files (images, CSVs, FITS, DICOM, audio, etc.), they are real files from real sources — NOT generated by scripts
- [ ] setup_task.sh does NOT contain any data generation code (no np.random, no faker, no astropy generation, no programmatic data fabrication)

### Design the Starting State
- [ ] Decide what messy/inconsistent state the task will start from (not just "find one entity to act on")
- [ ] For hard/very_hard tasks: multiple entities with errors, or data spread across features, requiring discovery
- [ ] Document all seeded state: what records exist, what errors are present, what the agent must find
- [ ] Verify all seeded data is present in the environment before finalizing

---

## Phase 2: Documentation

### Create README.md
- [ ] Task overview and domain context
- [ ] Goal: what the end state should look like (not how to get there)
- [ ] For hard/very_hard: do NOT include UI navigation steps or which feature to use
- [ ] For hard: include target identity and expected values; for very_hard: goal only
- [ ] Success criteria (what constitutes completion)
- [ ] Verification strategy (how each subtask is checked)
- [ ] Schema/data reference (relevant tables/fields for verifier implementation)
- [ ] Edge cases and potential issues

### Create task.json
- [ ] Unique task ID with version (`"id": "<task_name>@1"`)
- [ ] Clear, specific description — states the GOAL and end state, not the UI path
- [ ] If hard/very_hard: description does NOT spell out which menus/buttons/dialogs to use
- [ ] Difficulty rating (easy/medium/hard/very_hard)
- [ ] Appropriate timeout and max_steps inside `"init": { "timeout_sec": ..., "max_steps": ... }`
- [ ] **Both** hook paths set: `"pre_task"` (setup_task.sh) AND `"post_task"` (export_result.sh)
- [ ] Complete metadata with ground truth values
- [ ] Success spec pointing to verifier function: `"success": { "mode": "program", "spec": { "program": "verifier.py::<fn_name>" } }`
- [ ] **Verify format against an existing working task.json in the same environment** — open one of the existing tasks' task.json files and confirm your new file uses the same top-level keys (`id`, `version`, `env_id`, `init`, `hooks`, `metadata`, `success`)

## Phase 3: Implementation

### Create setup_task.sh
- [ ] Shebang: `#!/bin/bash`
- [ ] Source shared utilities
- [ ] Verify target entity exists (fail if not)
- [ ] Record initial counts to `/tmp/initial_*`
- [ ] Record timestamp to `/tmp/task_start_timestamp`
- [ ] Ensure application is running
- [ ] Take initial screenshot
- [ ] Print setup completion message

### Create export_result.sh
- [ ] Shebang: `#!/bin/bash`
- [ ] Source shared utilities
- [ ] Take final screenshot
- [ ] Read baseline from `/tmp/initial_*`
- [ ] Query current state
- [ ] Compare and calculate new counts
- [ ] Query for specific new records
- [ ] Validate values are in acceptable ranges
- [ ] Create JSON with all verification data
- [ ] Save to `/tmp/<task_name>_result.json`
- [ ] Print export completion message

### Create verifier.py
- [ ] Import required modules (json, tempfile, os, logging)
- [ ] Define verification function with correct signature
- [ ] Document scoring breakdown in docstring
- [ ] Get expected values from task_info metadata
- [ ] Copy result JSON from VM using copy_from_env
- [ ] **FIRST CHECK**: Wrong target = score 0, fail immediately
- [ ] Implement multi-criterion scoring (3+ criteria)
- [ ] Validate values are in realistic ranges
- [ ] Calculate total score
- [ ] Determine pass threshold
- [ ] Return structured result with feedback

### Set Permissions
```bash
chmod +x examples/<env_name>/tasks/<task_name>/*.sh
```
- [ ] All .sh files are executable (verify with `ls -la`)

## Phase 4: Testing and Validation

> **Clarification on what is required vs. recommended:**
>
> - **Required for every task:** Write the verifier, then run the **offline mock tests** (do-nothing, partial, wrong-target) described in `13_file_content_verification_and_offline_testing.md`. These run in seconds with no VM and catch the vast majority of scoring bugs.
> - **Recommended when practical:** Boot the environment and run the do-nothing test live. This catches script bugs (wrong paths, permission issues, missing data) that offline tests cannot. Do this at least once per environment, and ideally for the first task in a new environment.
> - **Not required:** Capturing screenshots, collecting evidence JSON, or saving artifacts to `evidence_docs/` for every single task. Evidence collection is valuable for debugging and review, but it is not a gate that blocks task completion.
>
> **Do not let testing/evidence phases prevent you from completing tasks.** If the offline mock tests pass and the scripts look correct, the task is ready. You can always come back to run live tests later.

### Offline Verifier Tests (Required)

These run in seconds with no VM — see `13_file_content_verification_and_offline_testing.md` for the full pattern:

- [ ] Do-nothing test: mock an empty/unchanged result → `passed=False`
- [ ] Wrong-target test: mock a result with the wrong entity → `score=0, passed=False`
- [ ] Partial test: mock a partially-complete result → partial score, `passed=False`

### Live Environment Test (Recommended)

Boot the environment and verify the scaffolding works end-to-end:

```python
env = from_config("examples/<env_name>", task_id="<task_name>")
obs = env.reset(seed=42, use_cache=False)
```
- [ ] Environment starts without errors
- [ ] Setup script runs (check `/tmp/initial_*` files exist)
- [ ] Application is running and accessible

```python
# Run do-nothing test live:
obs, reward, done, info = env.step([], mark_done=True)
result = info.get("verifier", {})
print(result)  # Should show score=0, passed=False
```

### Evidence Collection (Optional)

Useful for debugging and review, but not a blocking requirement:
- [ ] Screenshot saved to `examples/<env>/evidence_docs/`
- [ ] Evidence JSON created with database verification

### Full Completion Test — NOT REQUIRED

Do not design tasks to be solvable by the task creator — design them to be solvable by a capable agent. You do NOT need to personally complete the full task end-to-end. If the offline tests and live do-nothing test pass, the verifier is working.

```json

  "version": "1.0",
  "env_id": "<env_name>@0.1",
  "description": "...",
  "difficulty": "hard",
  "init": {"timeout_sec": 480, "max_steps": 60},
  "hooks": {
    "pre_task": "/workspace/tasks/<task_name>/setup_task.sh",
    "post_task": "/workspace/tasks/<task_name>/export_result.sh"
  },
  "metadata": {
    "target_id": 123,
    "target_name": "..."
```
