## Installation Quirks

1. **Docker pre-pull**: The `install_geoserver.sh` script pre-pulls both images during the `pre_start` hook. This avoids timeout during the `post_start` hook when Docker services start.

2. **Natural Earth data**: The install script downloads 4 Natural Earth 110m-scale shapefiles (countries, populated places, rivers, lakes) to `/home/ga/natural_earth/`. These are small files (~1MB total) suitable for fast import.

3. **GeoServer environment variables**: The Docker container accepts `GEOSERVER_ADMIN_USER` and `GEOSERVER_ADMIN_PASSWORD` to set admin credentials. We use `admin` / `Admin123!`. Also `SAMPLE_DATA=TRUE` loads GeoServer's built-in sample datasets (DEM, lakes, streams, etc.).

4. **ogr2ogr for shapefile import**: The kartoza/postgis container does NOT include `shp2pgsql`. Use `ogr2ogr` instead, which IS available at `/usr/bin/ogr2ogr`.

5. **PROMOTE_TO_MULTI for countries**: The Natural Earth countries shapefile contains mixed Polygon/MultiPolygon geometries. Without `-nlt PROMOTE_TO_MULTI`, ogr2ogr fails with "Geometry type (MultiPolygon) does not match column type (Polygon)".

## Service Timing

- PostGIS starts in ~10-20 seconds
- GeoServer (Tomcat) takes 60-120 seconds on first startup (WAR deployment + initialization)
- Total setup time: ~5-8 minutes (including Docker pull from cache and shapefile import)
- GeoServer readiness check: poll `http://localhost:8080/geoserver/web/` for HTTP 200/302

## PostGIS Authentication

- **CRITICAL**: Use `-h localhost` flag with `psql` to force TCP connection instead of Unix socket. Without it, peer authentication fails because the shell user doesn't match the PostgreSQL role.
- Always use `-e PGPASSWORD=geoserver123` with `docker exec` to provide the password.
- Database: `gis`, User: `geoserver`, Password: `geoserver123`
- PostGIS extensions `postgis` and `postgis_topology` are enabled automatically.

## GeoServer REST API

- Base URL: `http://localhost:8080/geoserver/rest`
- Authentication: `admin:Admin123!`
- Accept headers: `application/json` for data, `application/xml` for SLD styles

### Key Field Names (REST API vs GUI labels)

| REST API Field | GUI Label | Notes |
|----------------|-----------|-------|
| `maxRequestMemory` | Max rendering memory (KB) | NOT `maxRenderingMemory` |
| `maxRenderingTime` | Max rendering time (s) | Correct as-is |
| `watermark.enabled` | Enable watermark | Boolean field |

## Verification Pattern

1. **export_result.sh** runs inside the VM: queries GeoServer REST API, writes JSON to `/tmp/<task>_result.json`
2. **verifier.py** runs on the host: uses `copy_from_env()` to retrieve JSON, scores criteria
3. **Multi-level search**: exact name â†’ partial match â†’ newest entity. This handles agents that name things slightly differently.
4. **Integrity nonce**: Generated by `setup_task.sh`, stored in `/tmp/result_nonce`, verified by verifier.py

## File Permission Issues

- `/tmp/result_nonce` may be created by root (when pre_task runs as `sudo -E bash -lc ...`) but needs to be readable/writable by the `ga` user.
- Fix: `generate_result_nonce()` in task_utils.sh uses `rm -f` + `chmod 666` with sudo fallbacks.

## GeoServer Sample Data Layers

When `SAMPLE_DATA=TRUE`, GeoServer includes these layers in the `cite` workspace:
- DEM (raster elevation)
- Lakes (vector polygons)
- Streams (vector lines)
- Buildings, NamedPlaces, etc.

These are used by the `create_layer_group` task.
